# docs 執筆規範

本規範は `packages/docs` 配下の Markdown / MDX を執筆・推敲するときに従います。主に日本語ページを対象としますが、英語ページの構成・論証の運び方も同じ原則に従います(語彙・語順に関する規則は日本語のみです)。

日本語技術文書の一般規範(整形、文体、段落と論証の構成、論証の厳密さ、演出の抑制、LLM っぽい表現の禁止、冗長の排除、読者への誠実さ)を基層とし、その上に本 repo 固有の運用ルール(MDX、コードブロック、用語、日英併記、プラグイン README との分担)を重ねます。

## 文体

- 「です・ます」調を採用します。「〜だ」「〜である」で結ぶ常体は使いません。
- 体言止めを避け、動詞や形容詞などの述語で結びます。「〜が結論。」で終わらず、「〜が結論となります。」まで書き切ります。箇条書きの各項目でも同様に、名詞で終わらず述語まで書きます。
- 命令調の「〜せよ」「〜すること」は「〜します」の作業者判断形に置き換えます。ただし、コマンド手順やチェックリストなど、指示が読者の作業を直接駆動する箇所では「〜してください」の依頼形を使います。
- ですます調の中に、体言止めの見出しや引用文が混ざるのは許容します。あくまで地の文の語尾が対象です。

## 整形

- 一文ごとに改行します。段落の区切りは空行で示します。
- コード、差分、ログ、設定ファイルの断片はコードブロックで示します。
- 用語の由来や定式化の名称など、本筋から一段外れる補足は、本文に並べず脚注(`[^ラベル]`)に降ろします。
- 定義や分類の列挙は箇条書きで示してもかまいません。定義される用語は太字にします。
- 用語を本文中で初めて定義・導入するときも、その語は太字にします。すでに導入した語を話題として指すとき、引用、通称には「」を使い、太字と使い分けます(初出の定義は太字、以後の言及は「」)。
- ダッシュ(em ダッシュ `—`、horizontal bar `―`、いわゆる2倍ダッシュ「——」)は日本語の地の文・見出しで使いません。同格・補足の挿入(「A——挿入——B」)は括弧()に置き換え、言い換え・敷衍(「A——B」)は句点で二文に分けるか読点でつなぎます。範囲を示す en ダッシュ `–` や英語の複合語(`Curry–Howard` など)、コードブロック・書誌情報は対象外です。
- 中黒(・)は日本語の並列で使いません。ただし単一の固有名詞の内部では使ってもかまいません。
- 見出し・コラム見出しに、区切り線(罫線 `─` U+2500 やダッシュ類)で「種別──主題」「主題──概念」のように二要素を詰め込みません。見出しは単一の自然な句にします(要素を一つに絞るか、助詞・読点でつなぎます)。コラム見出しも「基礎」「補足」のような種別名だけで済ませず、内容を特定する句にします。
- 用語とその定義を並べる箇条書きは、区切り線ではなく全角コロンで「**用語**:説明」の形で書きます。

## 段落と論証の構成

パラグラフライティングを基本とします。段落は論証の一歩であり、読者が段落単位で論理を追える構成にします。

- 一つの段落には一つのトピックだけを置きます。場面の進行(調査、報告、検証、評価)が複数混ざった長い段落は、一歩ずつの段落に分割します。
- 段落の最初の文を読めば、その段落が何の話かわかるようにします。
- 段落の先頭では、前の段落との論理関係を接続表現で明示します(「であれば」「実際」「しかし」「この例自体からも」)。
- 新しい概念・術語を導入するときは、いきなり「XはYである」の辞書型断定で始めません。先に導入文で対象を置き、次にその働き・差分を述べ、必要なら三文目で定義を与えます。
- 論証は一方向に進めます。結論を出してから反論を処理し、結論を言い直す構成にしません。反論と疑念の処理を終えてから、結論を一度だけ置きます。
- 例への弁明(作為的に見える、への先回りなど)は、場面の山場の直後に挟んで流れを切りません。次の節の冒頭でまとめて処理します。
- 読者が立てそうな誤った解釈は、明示的に否定してから本当の理由を述べます(「その理由は『〜だから』ではありません。〜だからです」)。
- 「AではなくB」と否定するときは、否定の根拠を一文添えます。反実仮想(「もしAなら、〜だっただろう」)が使えることが多いです。
- 譲歩(「確かに〜」)では、事実の確認にとどめます。あとで訂正する内容を著者の声で因果として断定すると、自己矛盾になります。表面的な診断を一度認めたいときは、読者や通説の声に帰属させます(「〜と要約できてしまうかもしれない、と感じるかもしれません」)。
- 山場で効かせたい情報(数値、固有の事実)は、その手前の段落で先出ししません。
- 何かを否定・限定するときは、否定する命題そのものを「」で正確に書き出します。「何もかもが解決するわけではない」のような漠然とした否定で済ませません。
- 「別のページで扱う」のような前方参照は、論証が一段落した位置(段落末・節末)に置きます。論証の途中に挟んで流れを切りません。

docs 特有の追加ルールがあります。

- 各ページの冒頭には、そのページが答える問いを一段落で置きます。目次を読んで飛んできた読者が、そのページを読むべきかどうかをその段落で判断できるようにします。
- H2 セクションは論証の一段落と同じ役割を持ちます。セクションをまたぐときは、前セクションと何が変わったかを冒頭で示します。

## 論証の厳密さ

文章の論理にツッコミどころを残しません。書き上げたら、読み手の反論を先回りして次の点を点検します。

- 推量・可能性・読者の疑念・反実仮想として書かれている文を、機械的に断定へ変えません。
  「かもしれない」「だろう」「ようだ」「らしい」は、根拠なく主張を弱めている場合だけ削ります。
  事実未確認の可能性、ログからの推定、読者が抱きそうな疑念、反実仮想を表す場合は、その不確実性を保ちます。
  断定に直せるのは、本文内の根拠によって命題が確定している場合に限ります。
- 異なるものを「同じ」とまとめません。区別すべき対象(別々の決定、別々の原因、種類の違う問題)を一括りの言葉でくくりません。
- 複数の要因がある事象を、単一の原因に還元しません。例が複数種類の問題を含むなら、それぞれを切り分け、どの道具がどれを説明するのかを対応づけます。
- ページをまたいで、同じ概念の扱いを一致させます。あるページで「preset が担う」と分類したものを、別のページで「プラグインが担う」と書きません。分類・定義・用語の地位はサイト全体で揃えます。
- 因果を主張するときは、その機構(なぜそうなるのか)を一文で示します。「AだとBになる」とだけ書いて理由を省略しません。
- 検出・保証・解決を「必ず」できるかのように書きません。条件付きで正確に述べます(「〜しやすい」「〜できることが多い」「〜が成り立つときに限り」)。
- 主張は、挙げた例が実際にその全体を支えているかを確認します。例が主張の一部しか支えないなら、主張の範囲を例に合わせて狭めます。
- 「次節で扱う」と前方に逃がした論点は、本当にそこで回収されることを確認します。回収しない伏線を張りません。
- 譲歩や限定(「ただし」「とはいえ」)を置いたら、その後で必ず論を進めます。逆接で終えて宙吊りにしません。
- 節の中心となる語は、その節以前に定義・対象範囲を述べてから使います。定義せず使い始めません。
- 複数の概念を一つの上位語にまとめるときは、命名の直前に、それらが同じものに帰着すると一文で述べます。

## 読み手の負荷の管理

読者の記憶と注意は有限の資源として扱います。想定する読者は、Web 開発の一般的な語彙は持つが本ボイラープレートは初見の OSS ユーザーです。Eleventy や Preact の経験は前提としません。

- 後で参照する必要のない固有名(ファイル名、関数名、識別子)を出しません。「設定ファイル」「レンダリング用のプラグイン」のような一般的な言い方で済ませます。
- 抽象的な言い回しの指す内容が文脈から一意に決まらないときは、可能な場合は丸括弧による同格挿入でその場で特定し、できるだけ読者に前を読み返させないようにします。
- 新しい例や場面を追加して、読者が保持すべき文脈が増えるときは、前の例と何が違うのか、なぜもう一つ必要なのかを前置きして納得感を与えます。
- ページ冒頭や節の導入では、これから例で扱う内容に関係しない過剰な詳細を詰め込みません。
- 例の節の中でも、**その節の問い・帰結に関係しない**過剰な詳細だけを省きます。議論に必要な具体は残します。省略の典型は、装飾的精度(バージョン番号の細部、時刻、HTTP ステータスなど)や、後で参照しない固有名です。

## 視点と語り

- 例示では、結果の羅列や受動態(「特定され、判明した」)ではなく、行為者を主語にした動作の連なり(「設定ファイルを読み、該当箇所を書き換えます」)で書きます。
- 「入社2年目のエンジニアが」のような架空の人物設定を無意味に冠しません。
- 論証の中で読者を「あなた」と呼ばず、役割名(「開発者」「読者」「利用者」)で書きます。二人称の呼びかけは、場面への導入(「〜としましょう」)や章・本の結びなど、限られた要所にとどめます。
- ドキュメント自身を指すときは、「本ドキュメント」「本ボイラープレート」「本 preset」のように具体的に指します。「このプロジェクト」「うち」のような曖昧語や身内語は使いません。
- 対象を指す語は具体的に選びます。「AI」「ツール」のような広い語でぼかしません。
- 節で定式化・術語(Island、Partial Hydration など)を導入したら、以後はその語で通します。「機能」「仕組み」のような曖昧語に後退しません(定式化する前の導入語として「仕組み」などを使うのはかまいません)。
- 術語・訳語は、その分野で慣用されている語を選びます。意味の近い漢語を一般語の感覚で充てません。
- 人物そのものに言及するときは原綴りで書きます。ただし、歴史上の人物や、人名を冠した概念を定着名で紹介するときは、日本語で通用しているカタカナの通称を使います。
- 術語の響きを持つ語を、術語でない場面に流用しません。普通の言い方で書きます。

## 演出の抑制

演出の規範は全面禁止ではなく、節度の規範です。修辞は、それが効果を生む箇所でのみ使います。

- 溜め(「ここには〜が潜んでいる」)や修辞疑問で導出を演出するのは、緊張が議論に効く要所に限ります。説明で足りる箇所では、そのまま述べます。
- 短い決め台詞を独立した段落にして緊張を作る演出を多用しません。段落内の短い体言止めも、地の文の語尾では避けます(「文体」節の通り)。
- 本文中の太字強調を多用しません。誤読を防ぐ否定や節の帰結など、論理の要所に限り、一節に一、二箇所まで使ってもかまいません(導入部でも可)。それ以外は文の順序と構造で際立たせます。
- 「〜してはならない」という命令調の断定より、「〜するわけにはいきません」のような、作業者の判断として書く形を選びます。ただしコマンド手順やチェックリストなど、指示が読者の作業を直接駆動する箇所は「〜してください」の依頼形にします。
- 転回点を過剰に劇的にしません。事実を述べる一文で足りる場合が多いです。
- 帰結の列挙によって危険を煽りません。
- 「重要なのは〜である」のような前置きで主張を予告しません。主張をそのまま書きます。ただし、主張の様式を宣言する前置き(「標語として言い換えれば」など)は使ってもかまいません。
- 「AではなくBだった」という対句の決め台詞を多用しません。軽い補足や評価は括弧書きで添えてもかまいません。
- 慣用表現をひねった言い回しや、指す内容が一意に決まらない比喩を使いません。平易な動詞でそのまま言います。

## LLM っぽい表現の禁止

LLM が大量生成する、中身のない型に誘惑されません。書き上げたら、この節で点検します。本規範や本 repo の術語(preset、プラグイン、Island、Partial Hydration など)を議論に使うのは構いません。空虚な装飾として使うのが問題です。

次のような言い回しは、論点を増やさず「ちゃんと書いている感」だけを付ける LLM 口調です。使いません。

- **予告と総括**: 「重要なのは〜である」「本章では〜を扱う/探求する」「ここでは〜について見ていく」「まとめると」「要するに」(直前の言い換えだけのとき)、「〜に他ならない」
- **正面から系**: 「正面から扱う」「正面から回収する」「正面から見る/書く/立てる」——中身の代わりに姿勢だけを宣言するもの
- **空虚な形容**: 「不可欠」「核心的」「鍵となる」「根本的な」(主張の中身を説明せず強調だけするもの)、「多角的」「包括的」「総合的」(何をどう見たかを書かないもの)
- **空虚な動詞**: 「掘り下げる」「深掘りする」「言語化する」(何をどう書いたかを示さず終わるもの)、「触れる」「言及する」(一段落で済ませるだけのもの)
- **接続の型**: 「〜において」「〜という側面から」「〜の観点から」(新情報がないもの)、「さらに」「また」「加えて」の連打
- **弱い緩和と称賛**: 「〜と言えるだろう」「〜かもしれない」(根拠なく主張を弱める場合だけ。推量・仮定・読者の疑念・作中人物の認識なら残します)、「非常に」「極めて」「大いに」(中身のない強調)

悪い例: 「本ページでは、Partial Hydration の設計思想を正面から扱う」「多角的に分析すると、重要なのはインタラクティビティである」。
良い例: 「本ページでは、Partial Hydration の設計思想を扱います」「Partial Hydration の評価は、初期表示に必要な JS の量で決まります」。

## 冗長の排除

無駄な文章はなるべく残しません。

- 同じ主張を言い換えて繰り返しません。一つの主張は一度だけ書きます。
- 隣接する節が同じことを別の角度で述べているなら、役割が重複しています。片方に吸収して一つの節にまとめます。
- 場面を描写した直後に、その内容を要約し直しません。意味づけの一文だけを置きます。
- 同じ論理的役割を持つ並列の事実は、文を分けて重ねず一文にまとめます。その事実群の論理的地位は文頭の語で示します。
- 読者が自力で補える中間段階の説明は書きません。
- 数文にわたる議論を一文に圧縮できるなら、圧縮した一文だけを残します。要約の合図として「要するに」を使ってもかまいません。
- 接続や評価のためだけの文(「それ自体はよいことです」など)を置きません。
- 想像上の読者との問答(問いを立てて一語で答える形など)を修辞として使いません。主張はそのまま述べます。読者の反応を演じて応答する形も同様に避け、譲歩は地の文で簡潔に行います。
- 読者が抱きそうな発想を、メタな枠取り(「ここまでの話には自然な続きがあります」「〜という発想です」)で紹介しません。その発想自体を直接書きます。読者の疑問なら疑問文のまま書いてもかまいません。
- 「本ドキュメントもそれを否定しません」のような、著者の立場の弁明や断りを書きません。事実の記述だけを置きます。
- 文脈を最短で読み手と共有できる文章にします。導出を一歩ずつ展開しなくても伝わるなら、構造に名前を与えて言い切ります。
- 本文でまだ導入していない概念や文書名を、先回りして持ち出しません。
- ためらいのある弱い述語で済ませません。本文内の根拠で確定していることは強く具体的に言い切ります。ただし、不確実性・可能性・仮定・読者の疑念を表すための弱い述語は保持します。語調を整えるための意図的な緩和は許します。
- 文章のリズムを作るための接続表現(「しかし一方で」など)は、冗長と見なしません。

## 見出しの付け方

見出しは内容を特定できる具体的なものにします。その節が答える問い、または扱う対象を指す句にします。

- 作業の手順だけを述べる見出し(「例に戻す」「〜を読み直す」など)や、情報量のない見出しにしません。その節が答える問い、または扱う対象を見出しにします。
- 見出しを、節の結論を言い切る「セリフ」にしません。見出しの時点で読者がオチを知る状態を避けます。
- 節で扱っている対象を指す名詞句にしてもかまいません(見出しは体言止めを許します。「文体」節の「体言止めを避ける」は地の文の語尾に対する規則です)。
- 見出しが疑問形か断定形かは問いません。問うのは、扱う対象や読者の持つ問いを指しているかどうかです。

MDX での階層規約もあわせて示します。

- 一ページに H1 は一つだけ置きます。H1 はページの主題そのもので、frontmatter の `title` と一致させます。
- H2 はページを構成する論証の一歩に対応させます。サイドバーやページ内目次には H2 のみを出す設定を基本とします。
- H3 は H2 の中でさらに区切りたい要素を並べるときに使います。H3 は目次に出ないので、H3 の見出しだけで論を追える構成にしません。
- H4 以下は原則として使いません。使わなければ収まらない構造になっているときは、まずページ分割を検討します。
- サイドバー上の階層(getting-started > install など)と、そのページの H2 列は独立に決めてもかまいませんが、`title` はサイドバーの表示名と一致させます。

## 読者への誠実さ

- 例が作為的に見えうる場合、それを隠しません。読者の疑念を先回りして認め、現実に十分あり得ることの根拠を短く添えます。
- その根拠は、著者の断定ではなく、読者自身の経験に訴える一般的事実や通説に求めます。
- 確認していないことを、確認したかのように滑らかに書きません。

## MDX / JSX を混ぜる際の節度

MDX を採用しているのは、Island コンポーネントの実例をその場で動かすためであり、装飾のためではありません。

- ページ内で JSX を埋め込むのは、その節が説明している挙動を実際に動かして示す必要があるときに限ります。
- 埋め込む JSX は、そのページの説明対象に関連するものだけにします。ボタンやカードなどの汎用 UI を装飾目的で撒きません。
- Island(`.client.jsx` を伴うコンポーネント)を埋め込むときは、なぜ Island 化しているのかを本文で説明します。Island のコストを紹介せずに使うと、レシピの範を示す docs としては矛盾します。
- コンポーネントを新規に作るのは、同じパターンを 3 箇所以上で使うときに限ります。1 度きりの例なら本文中にインラインで書きます。
- インタラクティブでない静的な図表は Markdown の表・画像・箇条書きで済ませ、コンポーネント化しません。

## コードブロック / コマンド例の規範

- コマンド例は pnpm を前提とします。npm/yarn の等価コマンドは併記しません(必要であればレシピの別ページで扱います)。
- 実行例は Node 24 を前提とします。それ以外の Node バージョンでの挙動差に触れるときは、その差自体を主題とする節でのみ扱います。
- シェルセッションはプロンプト記号(`$`)を付けません。コマンドと出力を分けたいときは、コードブロックを二つに分けます。
- 実行結果の抜粋は、その節の議論に必要な行だけを残します。装飾的な精度(処理時間、パス、ハッシュ、色コードなど)は削ります。
- 差分は unified diff で示します。前後のコンテキストは 3 行を上限とします。省略は `// ...` ではなく `// (省略)` のように日本語で書きます。
- ファイル名を示す必要があるときは、コードブロック直前の一行に日本語で「`packages/docs/eleventy.config.js`」のようにパスだけを置きます。フェンス言語のあとに付ける方式(` ```js title="..." `)は採用しません。
- 環境変数、設定キーなどの識別子はインラインコード(`` `KEY_NAME` ``)で示します。

## 用語統一集

用語は次の対応で通します。ページ内で別の訳語を混ぜません。

| 日本語(本文で使う語) | 英語ページの対応語 | 補足                                                                               |
| -------------------- | ------------------ | ---------------------------------------------------------------------------------- |
| プリセット           | preset             | `@tuqulore-inc/eleventy-preset` を指す一般名詞として使います                       |
| プラグイン           | plugin             | Eleventy プラグイン全般を指します                                                  |
| Island               | Island             | 翻訳しません。「アイランド」「島」は使いません                                     |
| Partial Hydration    | Partial Hydration  | 翻訳しません。「部分ハイドレーション」は初出の説明でのみ併記できます               |
| MDX                  | MDX                | 翻訳しません                                                                       |
| ボイラープレート     | boilerplate        | 本 repo が提供するものを指す総称として使います                                     |
| テンプレート         | template           | scaffold で生成される初期プロジェクトを指します                                    |
| ページ               | page               | docs サイト内の 1 ページ、または Eleventy が出力する 1 URL を指します              |
| レシピ               | recipe             | 応用ケース章のエントリの単位を指します                                             |
| 開発者 / 利用者      | developer / user   | 本 repo を使ってサイトを作る側を「開発者」、そのサイトの読者を「利用者」と呼びます |

新しい術語を導入するときは、初出のページで英語表記を括弧で添えます(「**プリセット**(preset)」)。以後のページでは日本語のみで通します。

## 日英併記の運用

- 日本語版(`src/ja/`)を master、英語版(`src/en/`)を追随扱いにします。設計判断や新規コンテンツは常に日本語側で先に決めます。
- 英語版に翻訳が追いついていないページは、frontmatter の `translated: false` などで明示し、言語切替 UI 上は日本語版へフォールバックさせます。プレースホルダの英語ページを残して読者を袋小路に誘導しないようにします。
- 用語対応は上表を単一のソースとします。英訳を新たに決めるときは、上表を更新してから英語ページを書きます。
- 一方の言語だけで通じる比喩・故事・慣用句を使いません。使いたいときは意味を平易な語で言い直します。

## プラグイン README との相互参照ルール

各プラグインの README と docs は、内容の受け持ちを次のように分けます。

- **README が受け持つもの**: パッケージの一段落の概要、インストールコマンド、最小の Usage スニペット、API リファレンス(オプション一覧、型)。npm ページから読み取れる情報はここに残します。
- **docs が受け持つもの**: 設計思想、他プラグインとの関係、応用ケース、ロードマップ、ADR。ページを跨ぐ議論はすべて docs 側で行います。

docs のページに書きたい内容が「README の該当節の言い換え」で終わるなら、そのページを削って README へリンクだけ張ります。逆に README にコピペした説明が数段落を超えたら、README を短縮して docs 側へ本体を移します。

docs から README を参照するときは、その節の見出しではなくパッケージ名で指します(「`@tuqulore-inc/eleventy-plugin-preact` の README を参照します」)。README 側の見出し変更に追随できるようにするためです。
