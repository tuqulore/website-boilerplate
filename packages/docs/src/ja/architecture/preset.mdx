# プリセット

本ページは **プリセット** (`@tuqulore-inc/eleventy-preset`) の設計を扱います。プリセットが果たす役割と、この形を選んだ理由を示します。

## プリセットとは何か

プリセットは、Eleventy の設定を書くための入り口です。scaffold された `eleventy.config.js` は次の 1 行で済みます。

```js
import preset from "@tuqulore-inc/eleventy-preset";

export default preset();
```

内部でプリセットは、3 つの独立したプラグインを Eleventy に登録し、入力・出力ディレクトリ、Markdown 設定、`src/public/` の静的アセット、画像最適化を一括で有効にします。開発者は Eleventy の細部を触らずに、コンテンツ執筆と JSX/MDX テンプレートに集中できます。

## なぜプラグインを 3 つに分けたのか

プリセット本体はまとめ役に徹し、機能は独立したプラグインとして切り出しています。分割の対象は次の 3 つです。

- `@tuqulore-inc/eleventy-plugin-preact`: JSX/MDX のサーバサイドレンダリング
- `@tuqulore-inc/eleventy-plugin-preact-island`: Partial Hydration とクライアントバンドル
- `@tuqulore-inc/eleventy-plugin-postcss`: PostCSS 処理

分割の理由は 3 つあります。第一に、プリセットを介さずにプラグイン単体で導入したい既存の Eleventy プロジェクトを想定しています。プリセットに機能を丸ごと詰め込むと、その粒度での再利用ができません。

第二に、それぞれのプラグインが独立してテスト・リリースできる状態を保ちます。プラグインごとに `peerDependencies` と `engines.node` を宣言でき、破壊的変更をプラグイン単位で切り出せます。

第三に、Island を使わないサイトでは preact-island を外して差し替えることが原理的に可能です。プリセットは 3 プラグインを固定で読み込みますが、プラグインの API 面が独立しているため、将来プリセットを分岐させても影響範囲が限定されます。

## src/ と dist/ を固定にしている

プリセットの内部で `setInputDirectory("src")` と `setOutputDirectory("dist")` を呼んでおり、この 2 つのディレクトリは上書きできません。画像最適化や `src/public/` の静的アセットコピーが `src`・`dist` 前提で書かれているため、ここを可変にすると副作用が広がります。

「入力先を `pages/` にしたい」といった要望は、規約からの逸脱を許すよりも、プリセット自体をフォークするか、プラグインを直接組み合わせて別のプリセットを作る方が明快です。プリセットは「1 つの規約」を提供する層と割り切っています。

## extend で拡張する

プリセットは 1 引数の関数を受け取り、その関数を Eleventy 設定関数の末尾で呼びます。フィルタや追加プラグインを差し込みたいときに使います。

```js
import preset from "@tuqulore-inc/eleventy-preset";

export default preset((eleventyConfig) => {
  eleventyConfig.addFilter("myFilter", (value) => value);
});
```

extend の実行タイミングはプラグイン登録の後です。プリセットが登録した設定 (入出力ディレクトリ、プラグイン群) の上で、追加の設定を重ねる形になります。既存の設定をひっくり返す拡張は想定していません。

## 画像最適化と静的アセット

プリセットは 2 つの副作用を Eleventy build に足します。

- `src/**/*.{jpeg,jpg,png,webp,gif,tiff,avif,svg}` (ただし `src/public/` を除く) を `@11ty/eleventy-img` で最適化し、対応する `dist/` 配下へ書き出します。
- `src/public/` の中身を `dist/` 直下へ丸ごとコピーします。favicon、robots.txt、既に最適化済みの画像などをここに置きます。

`src/public/` は「そのまま出す」場所、それ以外の画像は「最適化を通す」場所という規約分けです。両者を混ぜないことで、意図しない再エンコードや品質劣化を避けられます。
